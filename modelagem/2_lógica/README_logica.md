## *Modelagem Lógica*

### *->* A modelagem lógica é o segundo passo na criação do banco de dados, nessa etapa, os relacionamentos entre as tabelas criados na etapa conceitual são aprofundados através da
utilização de chaves primárias e estrangeiras, que caracterizam a direção dessas ligações no qual algumas tabelas passam a receber informações de outras tabelas, tornando o banco
de dados cada vez mais íntegro.

### Desta forma, os seguintes modelos lógicos foram propostos:

*1_ACAD_HORA_PROF_MOD_MUSC* - Envolve os relacionamentos das tabelas: ACADEMIAS - HORARIOS - PROFESSORES - MODALIDADES: Cada academia tem um gerente, portanto, a tabela de academias
recebe o ID do Gerente como uma chave estrangeira. A entidade relacional NxNxNxN'acad_hora_prof_mod', agora em forma de tabela, é composta por quatro chaves estrangeiras originadas
nas tabelas academias, horarios, professores e modalidades, que juntas formam nesta tabela uma chave primária composta, impedindo a quebra de regras de negócio.

*2_CHECKINS_ALUNOS_ACAD* - Envolve os relacionamentos das tabelas: ACADEMIAS - CHECK_INS - ALUNOS - CHECKINS_CONTAGEM: Cada checkin registrado precisa armazenar o id da academia onde foi feito e do aluno que o fez e por este motivo, a tabela de checkins recebe esses dados como chaves estrangeiras. Nota-se que estas duas chaves não formam uma chave primária composta uma vez que o mesmo aluno pode registrar outros checkins porém na mesma academia. Por fim, a tabela de contagem de checkins recebe o id dos alunos como chave estrangeira e ao mesmo tempo, esse dado torna-se uma chave primária, permitindo que a unicidade das linhas seja mantida, visto o relacionamento 1x1 entre as tabelas de origem.

*3_TREINOS_OBJETIVOS_METAS* - Envolve os relacionamentos das tabelas: ACADEMIAS - TREINOS - MODALIDADES - ALUNOS - OBJETIVOS - METAS - METAS_ALUNOS_COMPLETAS: A tabela de treinos recebe como chaves estrangeiras o id do aluno, o id da modalidade e o id da academia, mas novamente, sem forma uma chave primária composta já que o aluno pode voltar a treinar a mesma modalidade na mesma academia. Na tabela de alunos, cada aluno pode registrar um objetivo que é recebido como chave estrangeira e além disso, as tabelas de metas e a contagem de metas recebem o id do aluno como chave estrangeira, com a particularidade da tabela de contagem utilizar o mesmo id como chave primária, mantendo a integridade do relacionamento 1x1 entre as tabelas de origem.

*4_MATRICULAS_PAGAMENTOS* - Envolve os relacionamentos das tabelas: MATRICULAS - PLANOS - ALUNOS - MODALIDADES E PAGAMENTOS: Cada matricula deve receber o id de um aluno, de uma modalidade e de um plano escolhido no momento do registro, esses dados são inseridos como chaves estrangeiras na tabela de matrículas. Por sua vez, o id gerado pelo novo registro de matricula é inserido na tabela de pagamentos como chave estrangeira assim que um pagamento novo é realizado pelo aluno.

*5_DESAFIOS_ALUNOS_MODALIDADES* - Envolve os relaciomentos das tabelas: DESAFIOS - DESAFIOS_TIPOS - ALUNOS E MODALIDADES: Neste grupo, cada desafio possui um tipo, portanto a tabela de desafios possui o id desse tipo como chave estrangeira. A entidade associativa, agora em forma de tabela, desafios_alunos recebem o id de um desafio e o id de um aluno enquanto a entidade/tabela desafios_modalidades_alunos_progressos recebe além destes, o id da modalidade. Nota-se que nessas duas tabelas, é preciso a criação de chaves compostas envolvendo os ids fornecidos para que o mesmo aluno não progrida no mesmo desafio em outra linha da tabela, o que causaria não conformidade com as regras do negócio.

*6_SORTEIOS_RECOMPENSAS_RESGATES* - Envolve os relacionamentos das tabelas: ALUNOS - PONTOS - RECOMPENSAS E SORTEIOS: Cada aluno possui um registro de pontuação unico na tabela de pontos, portanto esta tabela recebe o id do aluno como chave estrangeira e também o utiliza como chave primária para que seja mantido a unicidade proposta pelo relacionamento 1x1. Com os pontos, os alunos podem realizar resgates de recompensas, portanto ao ser realizado, a tabela de resgates recebe o id do aluno que fez um resgate bem como o da recompensa que foi resgatada, ambos dados como chaves estrangeiras. Nota-se que por não haver necesiddade de unicidade, o mesmo aluno pode resgatar a mesma recompensa diferentes vezes. Por fim, os sorteios, realizados esporadicamente recebem o id da recompensa e o id do aluno sorteado, ambos como chaves estrangeiras, novamente sem formar um par único na tabela.
